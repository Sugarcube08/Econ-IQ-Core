# EXPLAINABILITY CONTRACT (V2)

This document freezes the explainability framework for Econiq Core. All explanations, prediction key drivers (SHAP variables), and recommendation justifications are aligned exclusively with the 8 Canonical Scores, the 8 consolidated B2B dimensions, and verified feature definitions.

---

## 1. Traceability Principle

Every automated recommendation or model prediction generated by the backend must be accompanied by an explainability payload:

$$\text{Decision / Output} \longrightarrow \text{Key Driver Features} \longrightarrow \text{Supporting Dimensions} \longrightarrow \text{Traceable Rationale}$$

All legacy trust systems and rule engine explanations are removed.

---

## 2. SHAP Driver Mappings

When machine learning models run inference, they return a prioritized list of features in `key_drivers`. These feature names are mapped directly to user-facing explainability descriptors:

| Feature Name | Explainability Descriptor | Business Interpretation |
| :--- | :--- | :--- |
| `sales_window` | Sales Volume | Rolling 365d trade activity scale |
| `payments_window` | Payment Compliance | Outstanding invoice settlement volume |
| `returns_value_window` | Logistical Returns | Customer returns ratio impact |
| `credit_limit_window` | Credit Exposure | Total active outstanding debt utilization |
| `business_age_days` | Account Tenure | Total registered relationship longevity |
| `sales_recent` | Sales Velocity | Recent 20% window transaction trend |
| `active_duration_days` | Trading Consistency | Inter-event interval frequency |
| `participation_density` | Trading Day Density | Trading day frequency density in rolling window |
| `category_diversity_count`| Catalog Breadth | Number of unique product categories purchased |
| `product_diversity_count` | SKU Diversity | Number of unique product SKUs purchased |

---

## 3. Recommendation Rationales

Every generated recommendation is matched to a dynamic explanation block containing:
1. **Primary Score**: The core score triggering the recommendation (e.g. `trust_score`).
2. **Key Driver Dimension**: The specific consolidated dimension driving the score (e.g. `dim_discipline`).
3. **Rationale Template**: Structured markdown strings explaining the commercial context.

### Examples
* **Trigger Action**: `INCREASE_CREDIT_LIMIT`
  * **Primary Score**: `credit_score`
  * **Key Driver Dimension**: `dim_credit` (low utilization) and `dim_activity` (expanding sales)
  * **Rationale**: "Customer exhibits strong payment discipline (Discipline score $\ge 0.75$) with credit utilization below 30% and an expanding sales trajectory ($+15\%$ sales velocity)."
* **Trigger Action**: `TIGHTEN_PAYMENT_TERMS`
  * **Primary Score**: `risk_score`
  * **Key Driver Dimension**: `dim_discipline` (settlement delay)
  * **Rationale**: "Average payment delay has deteriorated to 14+ days past due (Discipline score $\le 0.40$), resulting in an elevated default risk."
* **Trigger Action**: `PROACTIVE_RETENTION`
  * **Primary Score**: `health_score`
  * **Key Driver Dimension**: `dim_stability` (inactivity) and `dim_relationship` (long tenure)
  * **Rationale**: "Customer has been inactive for an extended period (Health score $\le 0.45$), despite their high relationship tenure ($\ge 0.60$), signaling a high churn risk."
* **Trigger Action**: `ACCELERATED_COLLECTION`
  * **Primary Score**: `collection_score`
  * **Key Driver Dimension**: `dim_credit` (high outstanding) and `dim_discipline` (settlement delay)
  * **Rationale**: "Outstanding receivables are overdue and risk profile is elevated (Collection score $\le 0.50$), requiring immediate collection action."
